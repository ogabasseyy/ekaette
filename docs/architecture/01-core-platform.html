Security Considerations

Multiple ingress points exist across browser /ws, AT callbacks, WhatsApp webhooks, public APIs, and admin APIs. Each path needs independent authentication, authorization, and rate limiting.

SIP bridge to live voice service traffic relies on signed websocket tokens with explicit tenant/company/user claims, short expiry, and service-side validation. Service discovery stays explicit through environment-configured service URLs so rollback and region failover remain predictable.

Operational Considerations

The split architecture introduces coordinated deploy and rollback requirements across the main HTTP service, live voice service, and SIP bridge VM. Monitoring should treat the flow as distributed: ingress success, websocket session health, live-session latency, and callback/hangup correctness.

The SIP bridge adds an extra hop to the live voice service, so latency budgets must include VM → Cloud Run → Vertex AI, not only caller → webhook timing.
